Arkos's routing system is built on top of Express.js. If you've used Express before, most things will feel familiar. The main thing Arkos adds on top is auto-generated CRUD endpoints from your Prisma models — you don't write those routes at all. This page covers how to initialize your app and define your own custom routes.

## Setting Up Your App

<Tabs items={["v1.6+", "pre-1.6"]}>
  <Tab value="v1.6+">
```typescript title="src/app.ts"
import arkos from "arkos";

const app = arkos();

app.listen();
```
  </Tab>
  <Tab value="pre-1.6">
```typescript title="src/app.ts"
import arkos from "arkos";

arkos.init({});
```
  </Tab>
</Tabs>

### WebSockets and Custom Server Access

If you need access to the underlying HTTP server — for WebSockets, for example:

<Tabs items={["v1.6+", "pre-1.6"]}>
  <Tab value="v1.6+">

Build the app first with `app.build()`, then create the HTTP server manually and pass it to `app.listen()`:

```typescript title="src/app.ts"
import arkos from "arkos";
import http from "http";
import { Server } from "socket.io";

const app = arkos();

async function start() {
  await app.build();

  const server = http.createServer(app);

  const io = new Server(server);

  app.listen(server);
}

start();
```
  </Tab>
  <Tab value="pre-1.6">

Use the `configureServer` callback inside `arkos.init()`:

```typescript title="src/app.ts"
import arkos from "arkos";
import { Server } from "socket.io";

await arkos.init({
  configureServer: async (server) => {
    const io = new Server(server);
  },
});
```
  </Tab>
</Tabs>

<Callout type="info">
  `app.build()` handles all of Arkos's internal setup — loading Prisma, scanning
  modules, registering routes. It must be awaited before you create the HTTP server,
  otherwise built-in routes and middleware won't be in place when connections arrive.
  In the simple case (`app.listen()` with no arguments), this is done for you automatically.
</Callout>

### Configuring Express

<Tabs items={["v1.6+", "pre-1.6"]}>
  <Tab value="v1.6+">

In v1.6+, `arkos()` returns the Express app directly, so you can configure it before calling `app.listen()`:

```typescript title="src/app.ts"
import arkos from "arkos";
import helmet from "helmet";

const app = arkos();

app.set("trust proxy", true);
app.use(helmet());

app.listen();
```
  </Tab>
  <Tab value="pre-1.6">

Use the `configureApp` callback inside `arkos.init()`:

```typescript title="src/app.ts"
import arkos from "arkos";
import helmet from "helmet";

arkos.init({
  configureApp: async (app) => {
    app.set("trust proxy", true);
    app.use(helmet());

    app.locals.title = "My Arkos Application";
    app.set("view engine", "ejs");

    app.use((req, res, next) => {
      next();
    });
  },
});
```
  </Tab>
</Tabs>

### arkos.config.ts

Port, host, and other global settings live in your config file:

<Tabs items={["v1.6+ (defineConfig)", "pre-1.6 (plain export)"]}>
  <Tab value="v1.6+ (defineConfig)">
```typescript title="arkos.config.ts"
import { defineConfig } from "arkos";

const arkosConfig = defineConfig({
  globalPrefix: "/api",
  port: 3000,
  host: "localhost",
});

export default arkosConfig;
```
  </Tab>
  <Tab value="pre-1.6 (plain export)">
```typescript title="arkos.config.ts"
export default {
  port: 3000,
  host: "localhost",
};
```
  </Tab>
</Tabs>

<Callout type="info">
  `globalPrefix` is available since v1.6.0-beta. It defaults to `/api`. See
  [Configuration](/docs/getting-started/configuration) for all available options.
</Callout>

## Auto-Generated CRUD Routes

One of Arkos's core features is that it automatically generates full CRUD endpoints for every model in your Prisma schema — no route definitions needed. For a `Post` model, Arkos generates `GET /posts`, `POST /posts`, `GET /posts/:id`, `PATCH /posts/:id`, and `DELETE /posts/:id` out of the box.

See [Endpoints](/docs/core-concepts/generated-routes) for the full list of generated routes, their behavior, and how to customize or extend them.

## Custom Routes with ArkosRouter

For routes you write yourself, Arkos provides `ArkosRouter` — a thin wrapper around Express Router that adds declarative configuration support. The only difference from Express is that instead of passing a path string as the first argument, you pass a configuration object:

<Tabs items={["ArkosRouter", "Express Router"]}>
  <Tab value="ArkosRouter">
```typescript
import { ArkosRouter } from "arkos";

const router = ArkosRouter();

router.get({ path: "/reports" }, handler);
```
  </Tab>
  <Tab value="Express Router">
```typescript
import { Router } from "express";

const router = Router();

router.get("/reports", handler);
```
  </Tab>
</Tabs>

<Callout type="warn">
  Express Router is supported for backward compatibility but not recommended.
  Using it means you lose access to built-in authentication, validation, rate
  limiting, and other features that ArkosRouter provides through its
  configuration object.
</Callout>

That configuration object is where the real value lives. Authentication, validation, rate limiting and more — all declared alongside the route itself:

<Tabs items={["v1.6+", "pre-1.6"]}>
  <Tab value="v1.6+">
```typescript title="src/modules/reports/reports.router.ts"
import { ArkosRouter } from "arkos";
import reportsController from "@/src/modules/reports/reports.controller";
import reportsPolicy from "@/src/modules/reports/reports.policy";
import GenerateReportSchema from "@/src/modules/reports/schemas/generate-report.schema";

const reportsRouter = ArkosRouter();

reportsRouter.post(
  {
    path: "/reports/generate",
    authentication: reportsPolicy.Generate,
    validation: { body: GenerateReportSchema },
  },
  reportsController.generateReport
);

export default reportsRouter;
```
  </Tab>
  <Tab value="pre-1.6">
```typescript title="src/modules/reports/reports.router.ts"
import { ArkosRouter } from "arkos";
import reportsController from "@/src/modules/reports/reports.controller";
import { reportsAccessControl } from "@/src/modules/reports/reports.auth";
import GenerateReportSchema from "@/src/modules/reports/schemas/generate-report.schema";

const reportsRouter = ArkosRouter();

reportsRouter.post(
  {
    path: "/reports/generate",
    authentication: {
      resource: "report",
      action: "Generate",
      rule: reportsAccessControl.Generate,
    },
    validation: { body: GenerateReportSchema },
  },
  reportsController.generateReport
);

export default reportsRouter;
```
  </Tab>
</Tabs>

### Route Paths

<Tabs items={["v1.6+", "pre-1.6"]}>
  <Tab value="v1.6+">
Paths are defined relative to the `globalPrefix`. With `globalPrefix: "/api"` (the default), a route defined as `/reports/generate` is accessible at `/api/reports/generate`.
  </Tab>
  <Tab value="pre-1.6">
Paths are absolute. A route defined as `/reports/generate` is accessible at `/reports/generate` with no prefix applied automatically.
  </Tab>
</Tabs>

## Registering Routers

<Tabs items={["v1.6+", "pre-1.6"]}>
  <Tab value="v1.6+">

Pass your routers to `app.use()` before calling `app.listen()`:

```typescript title="src/app.ts"
import arkos from "arkos";
import reportsRouter from "@/src/modules/reports/reports.router";
import userRouter from "@/src/modules/user/user.router";

const app = arkos();

app.use(reportsRouter);
app.use(userRouter);

app.listen();
```
  </Tab>
  <Tab value="pre-1.6">

Pass your routers via the `use` array in `arkos.init()`:

```typescript title="src/app.ts"
import arkos from "arkos";
import reportsRouter from "@/src/modules/reports/reports.router";
import userRouter from "@/src/modules/user/user.router";

arkos.init({
  use: [reportsRouter, userRouter],
});
```
  </Tab>
</Tabs>

Routers registered this way are applied **before** Arkos's built-in routes, giving you full control over the request pipeline.

### Project Structure

Routers can live anywhere, but a consistent structure helps as your project grows. We recommend grouping everything under `src/modules/`:

```
src/
└── modules/
    ├── user/
    │   ├── user.router.ts        # Custom routes extending the User model
    │   ├── user.controller.ts
    │   ├── user.service.ts
    │   └── user.interceptor.ts  # Hooks into auto-generated User endpoints
    └── reports/
        ├── reports.router.ts     # Custom feature routes (no Prisma model)
        ├── reports.controller.ts
        └── reports.service.ts
```
