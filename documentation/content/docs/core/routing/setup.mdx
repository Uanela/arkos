Arkos's routing system is built directly on top of Express.js. If you've used Express before, everything here will feel familiar — the only meaningful difference is how you define route configuration.

## Setting Up Your App

```typescript title="src/app.ts"
import arkos from "arkos";

const app = arkos();

app.listen();
```

If you need access to the underlying HTTP server — for WebSockets, for example — use `createServer` from Arkos instead of Node's `http.createServer`. This ensures Arkos can finish its internal setup correctly:

```typescript title="src/app.ts"
import arkos from "arkos";
import http from "http";
import { Server } from "socket.io";
import arkosConfig from "#/arkos.config"

const app = arkos();

async function bootstrap(){
  await app.init()

  const server = http.createServer(app);
  const io = new Server(server);

  const { port, host } = arkosConfig
  server.listen(port, host);
}

bootstrap()
```

<Callout type="info">
  `app.listen()` and `server.listen()` only accept an optional callback. Port
  and host are configured in `arkos.config.ts` and managed by Arkos — passing
  them directly to `listen()` is not supported to avoid mismatches with your
  config.
</Callout>

### arkos.config.ts

Port, host, and other global settings live in your config file:

```typescript title="arkos.config.ts"
import { defineConfig } from "arkos";

const arkosConfig = defineConfig({
  globalPrefix: "/api",
  port: 3000,
  host: "localhost",
});

export default arkosConfig;
```

## Two Types of Routes

Arkos has two categories of routes:

**Built-in routes** are automatically provided by the framework — CRUD endpoints generated from your Prisma models, authentication endpoints, and file upload endpoints. You don't write these, but you can intercept and customize them. These are covered in [Endpoints](/docs/routing/endpoints) and [Interceptors](/docs/routing/interceptors).

**Routers** are routes you write yourself, exactly like you would in a regular Express app. This is what the rest of this section focuses on.

## ArkosRouter

For your routes, Arkos provides `ArkosRouter` — a thin wrapper around Express Router that adds declarative configuration support. The only difference from Express is that instead of passing a path string as the first argument, you pass a configuration object:

<Tabs items={["ArkosRouter", "Express Router"]}>
  <Tab value="ArkosRouter">
```typescript
import { ArkosRouter } from "arkos";

const router = ArkosRouter();

router.get({ path: "/reports" }, handler);
```
  </Tab>
  <Tab value="Express Router">
```typescript
import { Router } from "express";

const router = Router();

router.get("/reports", handler);
```
  </Tab>
</Tabs>

<Callout type="warn">
  Express Router is supported for backward compatibility but not recommended.
  Using it means you lose access to built-in authentication, validation, rate
  limiting, and other features that ArkosRouter provides through its
  configuration object.
</Callout>

That configuration object is where the real value lives. Authentication, validation, rate limiting and more — all declared alongside the route itself:

```typescript title="src/modules/reports/reports.router.ts"
import { ArkosRouter } from "arkos";
import z from "zod";
import reportsController from "./reports.controller";

const reportsRouter = ArkosRouter();

reportsRouter.post(
  {
    path: "/reports/generate",
    authentication: {
      resource: "report",
      action: "Generate",
      rule: ["Admin", "Manager"],
    },
    validation: {
      body: z.object({
        type: z.enum(["sales", "inventory"]),
        startDate: z.string().datetime(),
        endDate: z.string().datetime(),
      }),
    },
    rateLimit: {
      windowMs: 60_000,
      max: 10,
    },
  },
  reportsController.generateReport
);

export default reportsRouter;
```

Everything else — middleware chaining, `req`, `res`, `next`, async handlers — works exactly like Express.

### Route Paths

Paths are defined relative to the `globalPrefix`. With the default `globalPrefix: "/api"`, a route defined as `/reports/generate` is accessible at `/api/reports/generate`.

<Callout type="info">
  The global prefix is configured in `arkos.config.ts` via the `globalPrefix`
  option. It defaults to `/api`. See [Configuration](/docs/getting-started/configuration) for more.
</Callout>

## Registering Routers

Pass your routers to `app.use()` before calling `app.listen()`. Routers and middleware registered this way are applied **before** Arkos's built-in routes, giving you full control over the request pipeline:

```typescript title="src/app.ts"
import arkos from "arkos";
import reportsRouter from "./modules/reports/reports.router";
import userRouter from "./modules/user/user.router";

const app = arkos();

app.use(reportsRouter);
app.use(userRouter);

app.listen();
```

### Project Structure

Routers can live anywhere, but a consistent structure helps as your project grows. We recommend grouping everything under `src/modules/` — both Prisma model-related code and custom feature routers:

```
src/
└── modules/
    ├── user/
    │   ├── user.router.ts        # Custom routes extending the User model
    │   ├── user.controller.ts
    │   ├── user.service.ts
    │   └── user.interceptor.ts  # Hooks into auto-generated User endpoints
    └── reports/
        ├── reports.router.ts     # Custom feature routes (no Prisma model)
        ├── reports.controller.ts
        └── reports.service.ts
```
